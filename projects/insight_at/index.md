# InsightAT（v0.1）：面向系统落地的 CLI 优先增量式 SfM 重建算法

**InsightAT** 是一套开源、**高度自动化**的**增量式**运动恢复结构（Structure-from-Motion）系统。目标是从图像到高质量**稀疏重建**的**一条默认命令**即可完成，更强调**工程鲁棒性与易用性**，而不是把大量调参面暴露给非专家用户。

项目尝试用更**系统化**的视角做 SfM：用**缺省可跑通**的端到端流水线覆盖常见采集场景，在**少调参**的前提下，通过 **CLI 优先**的架构保持**可脚本、无头、上云**与批处理友好。

## 设计侧重

- **端到端自动化（图像 → 稀疏模型）** 作为主线用户路径  
- **GPU 加速** 的特征提取与匹配，基于成熟实现（具体后端以你的构建与配置为准）  
- **异步 I/O**，在大图集、重 GPU 阶段上提升有效吞吐  
- **CLI 模块化**：各步骤为独立可执行文件，便于批跑、封装与容器化部署  
- **混合式增量策略（当前设计思想）**  
  - **光束法平差（BA）** 在**保留状态子集**上工作，使模型变大时仍保持**可算、可控的成本**。  
  - **新视图定位 / 重投影 / 位姿拉入（resection/registration 相关）** 使用**完整 track 结构**，以保留强几何约束、提升**鲁棒性**。  
  换言之：**用子集 BA 换效率，用全量 track 做 resection 换稳定**，在不同阶段做分工。  

## 路线图：层次化 SfM 为长期目标

**层次化（多层级、全局可伸缩编排的）SfM** 是**长期目标**：希望在同一套系统里同时走向 **鲁棒、快速、真正意义的大规模覆盖**，而不要求用户用手工切块、手工拼接来「替系统分层」。

**v0.1 尚未提供** 多层级全局调度或分布式层次化 SfM。当前发布的是**单条增量式管线**，但已经体现**子集 BA + 全量 track 做 resection** 的设计，可视作向该目标迈进的**中间形态**。

## 实践中在验证什么

仓库内提供 **可脚本化基准工具**（例如类 ETH3D 的数据准备、与常见稀疏 SfM 基线的对比、以及在可用时使用公开标定或 GT 做几何一致性检查），以支持**可复现**的对照。文档中也提供基于 **Docker 的 Linux 构建** 路径，降低环境差异带来的摩擦。

*说明：* 这类对比针对的是**同一数据布局、同一评价脚本上的端到端稀疏重建**；并不声称内部每一个模块与某一第三方实现**逐行或逐算法等价**（尤其在特征与匹配实现上）。

## 当前适用场景

- 强调吞吐与**流程自动化**的**大体量图像**（如无人机/航空）——仍受单机显存、分辨率与算力限制  
- **服务器 / 云端**、**批处理** 与**无头**运行（脚本化、可容器化）  
- **下游衔接**：**MVS**、**3D 高斯泼溅** 等能消费导出稀疏模型或中间结果的管线（具体格式以仓库说明为准）  

## v0.1 范围说明（早期版本）

- **API、数据格式与内部实现细节** 仍可能**演进**  
- **层次化 / 分布式全局 SfM** **不在本版交付范围**；本版是**单引擎增量式**实现，但已明确体现 **子集 BA + 全量 track resection** 的设计取向  
- **主要验证环境** 为 **Linux**；v0.1 不以多平台为承诺  
- 「**大规模**」应理解为：在**单节点、单管线工程实现**下能扩展到的范围；**不是**对「星球级、无界单机、全自动切块合并」的承诺  

## 希望收到的反馈

越具体越有帮助：数据类型、张数、典型分辨率、**GPU/显存**、**Docker 或原生**、以及失败时的**简短日志**。我们特别关心：

- **默认可用性**：在「不额外调参」时，**最先**在什么数据上、以什么方式失败  
- **架构与 CLI 边界**：在真实工作流中集成、二次开发时的摩擦点  
- **性能与失败模式**：差初始化、track 过稀、显存 OOM、注册不稳定等
